# Rx vs Coroutines Simple Server

Учебный backend-проект на `Spring Boot` и `Kotlin`, который используется как серверная часть для проекта `Rx_vs_Cor_SimpleClient`.

Сервер специально сделан максимально простым, чтобы основное внимание можно было сосредоточить не на backend-логике, а на изучении клиентских технологий:

- `RxJava`
- `Kotlin Coroutines`
- в перспективе `Kotlin Flow`

Проект предоставляет небольшой HTTP API для отправки и получения списка студентов. Этого достаточно, чтобы на Android-клиенте сравнивать разные подходы к асинхронной работе с сетью на одной и той же задаче.

## Цель проекта

Основная задача проекта — быть простым учебным сервером для клиентского приложения, в котором сравниваются:

- выполнение сетевых запросов через `RxJava`
- выполнение сетевых запросов через `Kotlin Coroutines`
- возможное развитие в сторону `Flow`

Смысл этого backend-проекта не в сложной бизнес-логике, а в том, чтобы:

- дать клиенту стабильные HTTP-эндпоинты
- упростить отладку сетевого взаимодействия
- показывать одинаковый сценарий работы для разных клиентских реализаций
- не перегружать учебный стенд лишней архитектурой

## Что реализовано в проекте

- запуск HTTP-сервера на `Spring Boot`
- прием данных студента через `POST`
- получение списка студентов через `GET`
- хранение студентов в памяти приложения без базы данных
- сериализация JSON через `Jackson`

## Роль в связке с клиентом

Этот проект является серверной частью для учебного Android-клиента. Клиентское приложение отправляет данные студента на сервер и получает список студентов обратно, а затем использует эти запросы для демонстрации различий между `RxJava` и `Kotlin Coroutines`.

Иными словами:

- клиент нужен для изучения подходов к асинхронной работе
- сервер нужен как простой источник и приемник данных
- вместе они образуют небольшой учебный стенд для практики сетевого взаимодействия

## API

### Добавить студента

`POST /names/players/add`

Пример JSON:

```json
{
  "name": "Alex",
  "lastName": "Ivanov",
  "firstName": "Alex",
  "age": 20,
  "course": 2,
  "isPass": true
}
```

Что делает эндпоинт:

- принимает объект `Student`
- добавляет его в список в памяти
- возвращает сохраненный объект в ответе

### Получить список студентов

`GET /names/players/list`

Что делает эндпоинт:

- возвращает текущий список студентов
- список хранится только в памяти приложения
- после перезапуска сервера данные очищаются

## Модель данных

В проекте используется модель `Student` со следующими полями:

- `name`
- `lastName`
- `firstName`
- `age`
- `course`
- `isPass`

Модель намеренно простая, потому что ее основная задача — быть удобным payload для клиентских HTTP-запросов.

## Использованные технологии

- Kotlin
- Spring Boot 2
- Spring Web
- Maven
- Jackson Kotlin Module

## Структура проекта

- `UserSaverApplication.kt` — точка входа в `Spring Boot` приложение
- `NameController.kt` — REST-контроллер с эндпоинтами добавления и получения студентов
- `Student.kt` — модель данных студента
- `application.properties` — файл конфигурации приложения

## Как запустить проект

### Требования

- `JDK 8` или выше
- доступ к терминалу

### Запуск из терминала

Windows:

```powershell
.\mvnw.cmd spring-boot:run
```

Linux/macOS:

```bash
./mvnw spring-boot:run
```

По умолчанию приложение стартует на порту `8080`.

После запуска сервер будет доступен по адресу:

```text
http://localhost:8080/
```

## Как проверить API

### Добавление студента

```powershell
curl -X POST http://localhost:8080/names/players/add `
  -H "Content-Type: application/json" `
  -d "{\"name\":\"Alex\",\"lastName\":\"Ivanov\",\"firstName\":\"Alex\",\"age\":20,\"course\":2,\"isPass\":true}"
```

### Получение списка студентов

```powershell
curl http://localhost:8080/names/players/list
```

## Как связать с Android-клиентом

1. Запустите этот сервер.
2. Проверьте IP-адрес машины, на которой он работает.
3. Укажите этот адрес в клиентском проекте в поле базового URL.

Если клиент запускается на реальном Android-устройстве, в базовом URL нужно указывать не `localhost`, а IP компьютера в локальной сети.

Пример:

```kotlin
val URL = "http://192.168.1.67:8080/"
```

## Ограничения текущей реализации

- данные хранятся только в памяти
- база данных не используется
- нет валидации входных данных
- нет обработки ошибок и отдельного сервисного слоя
- контроллер сделан максимально просто, чтобы не отвлекать от клиентской части

Это нормальное решение для учебного проекта, где сервер нужен прежде всего как тестовый HTTP API для Android-клиента.

## Возможные улучшения

- добавить постоянное хранение данных через базу данных
- вынести бизнес-логику в service-слой
- добавить DTO и валидацию запросов
- настроить `application.properties`
- покрыть API тестами
- добавить `CORS`, если сервер будет использоваться не только Android-клиентом

## Итог

Этот проект полезен как простой серверный стенд для изучения клиентских технологий. Он дает минимальный, понятный и рабочий HTTP API, на котором удобно сравнивать, как один и тот же сетевой сценарий реализуется через `RxJava` и `Kotlin Coroutines` в Android-приложении.
